# 代码质量审查记录

本记录按 Client 与 Runtime 两个产品模块持续更新；Runtime SDK 是 Runtime 对外协议子包。目标不是整理格式，而是消除会造成数据损坏、重复执行、错误误判和恢复失败的问题。

## 审查方法

- 从公开入口向状态所有者追踪，不以单个报错位置为边界。
- SQL 重点检查事务原子性、外键、唯一约束、并发写入、迁移和失败回滚。
- 命令重点检查不可变载荷、稳定命令编号、重复投递和冲突响应。
- 错误重点检查分类是否跨 Runtime、SDK 和 App 保留，而不是只剩错误文案。
- 每项修复先增加能复现问题的测试，再执行模块测试；最终统一执行全量测试和 E2E。

## 模块边界与主链路

- Client：负责用户操作、尚未接纳命令和 Runtime 状态的本地投影；不能成为 Run、对话或工具结果的权威来源。
- Runtime SDK：只负责 HTTP、SSE、命令回执、快照和错误的无损协议翻译；不保存产品状态，也不包含 Electron 或界面逻辑。
- Runtime：负责命令日志、SQLite 状态、作业租约、Agent 执行、模型与工具账本、检查点及终态提交。
- 主链路：Client 生成稳定命令编号，经 SDK 发送；Runtime 在一个事务内接纳命令和作业；租约拥有者执行 Agent；模型和工具先写账本再产生外部效果；结果原子写入后，SDK 通过 SSE 和快照把权威状态投影回 Client。

## 已确认并修复

### 1. SQLite 外键只声明、未执行

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P2 幂等命令、P4 快照、P10 等待决定、P12 结果提交
- 状态所有者：`LocalStore`
- 旧路径：主连接和各个临时事务连接只配置行类型与忙等待，没有逐连接开启 `foreign_keys`。
- 风险：SQLite 默认关闭外键检查，孤立事件、作业、线程项或工具记录可能成功写入，后续只能表现为难以恢复的投影错误。
- 修复：所有 Runtime SQLite 连接统一执行相同连接配置并开启外键检查。
- 证明：新增测试分别从主连接和独立事务连接写入孤立事件，均必须触发 `IntegrityError`；相关作业和结果提交测试通过。

### 2. SSE 握手错误丢失结构化信息

- 模块：Runtime SDK
- 主阶段：P4 客户端读取快照并订阅变化
- 相邻阶段：P3.6 命令回执、P12 客户端收敛终态
- 状态所有者：Runtime HTTP 响应；SDK 负责无损翻译
- 旧路径：普通 HTTP 接口抛出带 `status/code` 的 `RuntimeHTTPError`，SSE 接口却重新抛出普通 `Error`。
- 风险：App 无法区分鉴权失败、Run 不存在、游标冲突和服务端错误，只能依赖易变化的错误文案。
- 修复：SSE 的非游标错误统一保留 HTTP 状态和 Runtime 错误码；游标重置仍使用专用错误类型。
- 证明：新增 SDK 测试验证 404、`run_not_found` 和原始消息均被保留。

### 3. 定时任务领取不是原子操作

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P5 作业领取、P12 结果提交
- 状态所有者：`LocalStore`
- 旧路径：先查询到期任务，再更新状态，最后逐条重新读取；查询与更新之间没有数据库锁。
- 风险：两个 Runtime 或恢复进程可能同时拿到同一任务并各自启动 Run。稳定命令编号能防止部分重复，但不能替代任务领取本身的所有权。
- 修复：在独立连接的 `BEGIN IMMEDIATE` 事务中完成查询和状态更新，提交后只返回本事务领取的记录。
- 证明：新增两个 `LocalStore` 并发领取同一任务的测试，合并结果只能出现一次任务编号；定时任务测试全部通过。

### 4. 永久拒绝的待发送命令被无限重试

- 模块：Client、Runtime SDK
- 主阶段：P2 客户端提交幂等命令
- 相邻阶段：P3 命令接纳、P4 客户端投影
- 状态所有者：Client 只拥有尚未被 Runtime 接纳的本地命令；Runtime 接纳后立即成为权威状态。
- 旧路径：SDK 吞掉每条命令的错误并只返回成功数量；App 只要数量不足，就每两秒重试整个未完成集合。
- 风险：工作区撤销、命令冲突、校验失败等永久 4xx 会产生无限请求；本地助手消息一直显示等待，用户也看不到真正原因。
- 修复：SDK 返回逐条失败及可重试分类；App 只自动重试网络、超时、限流和 5xx。永久拒绝会原子删除待发送命令并把本地助手投影结算为失败，同时保留其他线程继续投递。
- 证明：SDK 测试覆盖 409 与另一线程并行成功；IndexedDB 测试验证命令删除和失败投影在同一事务完成；App 相关测试与生产构建通过。

### 5. 请求校验错误被降级为通用 422

- 模块：Runtime SDK
- 主阶段：P3.2 请求校验
- 相邻阶段：P2 命令提交、P3.6 命令回执
- 状态所有者：Runtime 负责安全清洗校验错误；SDK 负责无损显示清洗后的结果。
- 旧路径：Runtime 返回 FastAPI 的安全错误数组，SDK 只识别字符串或对象，因此丢弃字段位置和消息。
- 风险：用户只看到 `Runtime HTTP 422`，无法知道哪个设置错误；上层只能再次自行解析响应，形成重复错误封装。
- 修复：SDK 识别清洗后的第一条字段错误，组合字段位置和消息；仍不读取或显示被 Runtime 删除的 `input` 和 `ctx`。
- 证明：SDK 测试验证状态码和 `body.memory` 错误消息均被保留。

### 6. 模型编号的客户端与服务端校验过宽

- 模块：Client、Runtime SDK、Runtime
- 主阶段：P3.2 请求校验
- 相邻阶段：P2 模型选择、P5 执行准备
- 状态所有者：Runtime API 模型；SDK 对本地缓存值做同等的提前校验。
- 旧路径：`local:<供应商>:<模型>` 只检查冒号位置，供应商和模型中包含空白仍会通过，直到执行准备阶段才以“模型不存在”失败。
- 风险：损坏的本地缓存或手工输入会被当作有效选择，并生成本可在信任边界直接拒绝的持久命令。
- 修复：Runtime 和 SDK 统一要求供应商与模型部分都不含空白，同时保留模型编号内部的冒号。
- 证明：SDK 与 Runtime HTTP 测试均覆盖空白编号；通过 `sync-schemas` 重新生成 OpenAPI，线上的两个请求字段约束已同步，TypeScript 构建检查通过。

### 7. Skill 与 MCP 配置写入可能损坏或互相争用

- 模块：Runtime
- 主阶段：P6 绑定资源并取得 Agent 定义
- 相邻阶段：P3 配置请求接纳、P8 工具可见性
- 状态所有者：Runtime 管理的本地 Skill 与 MCP 配置文件。
- 旧路径：Skill 直接覆盖 `SKILL.md`；MCP 虽然先写临时文件，但所有请求共用同一个 `.tmp` 名称。
- 风险：写入中断会留下半个 Skill 文件；并发 MCP 请求可能互相覆盖临时文件或使其中一个替换失败。
- 修复：两者复用唯一临时文件、刷新文件内容后执行操作系统原子替换；失败时清理临时文件并保留旧文件。
- 证明：新增替换失败测试，确认旧内容不变且临时文件被清理；Skill 与 MCP 的 59 项测试全部通过。

### 8. SSE 合法 JSON 可以绕过事件协议

- 模块：Runtime SDK
- 主阶段：P4 客户端读取快照并订阅变化
- 相邻阶段：P8 事件翻译、P12 客户端投影
- 状态所有者：Runtime 生成事件；SDK 验证公开事件信封。
- 旧路径：SDK 只执行 `JSON.parse` 后强制转换类型，不验证 `event_type`。
- 风险：Runtime 字段漂移或代理返回错误 JSON 时，App 的事件分支全部无声跳过，Run 看似一直运行且没有明确协议错误。
- 修复：SDK 在 SSE 边界要求非空字符串 `event_type`，不合法信封进入既有的 `Malformed Agent SSE` 错误路径。
- 证明：新增合法 JSON 但缺少事件类型的测试；SDK 测试与构建通过。

### 9. 主 SQLite 连接在写入失败后遗留事务

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P3.4 接纳检查、P3.6 命令回执
- 状态所有者：`LocalStore`
- 旧路径：主连接使用 SQLite 隐式事务，单条写入遇到约束或触发器错误时没有统一回滚；后续独立写事务可能被这个连接自己阻塞。
- 风险：一次设置、供应商、工作区或计划任务写入失败后，Runtime 仍然存活，但后续事件和命令可能持续报数据库锁定。
- 修复：主连接启用自动提交；真正需要多语句原子性的路径继续使用已有显式事务。启动迁移明确包在 `BEGIN IMMEDIATE` 中，失败时回滚并关闭连接。
- 证明：新增触发器强制设置写入失败，确认连接不遗留事务且随后仍可从独立连接追加事件。

### 10. Run 重放在幂等检查前访问当前模型状态

- 模块：Runtime
- 主阶段：P3.3 幂等检查
- 相邻阶段：P3.2 请求校验、P3.4 接纳检查
- 状态所有者：持久命令日志
- 旧路径：`start_run()` 先读取模型供应商、凭据和附件状态，再由 Store 查询命令日志。
- 风险：已经成功接纳的命令在重放时仍受当前凭据库、供应商配置和文件系统延迟影响，不能稳定地立即返回原回执。
- 修复：先用规范化后的不可变命令载荷查询原 Run；未命中后才执行当前协议、能力、附件和模型接纳检查。事务内仍保留第二次幂等检查以关闭并发窗口。
- 证明：重放测试把模型接纳函数替换为必定失败的哨兵，仍能取得原 Run；不同载荷继续返回冲突。

### 11. 定时任务的恢复和终态可以被迟到写入覆盖

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P5 作业领取、P12 结果提交
- 状态所有者：`local_scheduled_runs`
- 旧路径：计划从 `scheduled` 领取为 `running` 后、Run 编号写回前若崩溃，重启会直接判失败；绑定 Run、取消、完成和通知更新也没有状态前置条件。
- 风险：合法的计划任务因崩溃丢失；迟到的恢复进程可替换 Run 编号或把已完成状态改成失败；取消与领取并发时可能显示已取消但仍执行；未完成计划可被提前标为已通知。
- 修复：缺少 Run 编号的 `running` 计划重新走稳定命令编号的幂等启动路径；绑定、取消、完成和通知均改为带状态条件的单条更新，终态和首次通知不可覆盖。
- 证明：测试覆盖领取后崩溃恢复、终态与 Run 绑定不可变、通知只能在完成或失败后写入且重复调用不改变时间，以及两个 Store 不能重复领取。

### 12. 相同产物编号可以静默接受不同内容

- 模块：Runtime
- 主阶段：P10 工具执行与回执
- 相邻阶段：P11 执行结算、P12 结果提交
- 状态所有者：`local_artifacts`
- 旧路径：显式 `artifact_id` 已存在时直接返回旧记录，不比较内容、工具来源和元数据。
- 风险：工具重试若复用编号却产生不同结果，Runtime 会把旧产物误当成新结果，破坏回执与实际副作用的一致性。
- 修复：产物不可变字段完全相同才返回原记录；同编号不同内容抛出不可重试的 `ArtifactConflictError`，原记录保持不变。
- 证明：测试覆盖相同内容重放成功、不同内容明确冲突和旧内容不被覆盖。

### 13. 相同供应商配置会轮换凭据并撤销排队任务

- 模块：Runtime
- 主阶段：P3.4 接纳检查
- 相邻阶段：P5 执行准备、P6 模型绑定
- 状态所有者：供应商配置表和系统凭据库
- 旧路径：每次 `PUT` 只要携带 API Key 就生成新凭据引用，SQL 也无条件递增供应商版本。
- 风险：客户端因响应丢失重试完全相同的保存请求，会使已冻结旧版本绑定的排队 Run 被误判为配置已撤销。
- 修复：相同密钥复用现有凭据引用；供应商配置只有字段实际变化时才更新并递增版本，模型 JSON 使用稳定序列化。
- 证明：重复提交同一供应商后，完整数据库记录、版本、更新时间和凭据引用均保持不变；真实配置变化仍会递增版本并使旧绑定失效。

### 14. 相同 Runtime 设置补丁无条件递增版本

- 模块：Runtime
- 主阶段：P3.4 接纳检查
- 相邻阶段：P5 执行上下文冻结
- 状态所有者：`local_runtime_settings`
- 旧路径：`json_patch` 即使没有改变最终 JSON，也会更新版本和时间。
- 风险：重复保存产生虚假的配置变更和不必要的 WAL 写入，客户端无法用版本区分真正变化。
- 修复：仍使用单条 SQLite `json_patch`，但仅在结果 JSON 与当前值不同的时候更新；无变化直接返回现有记录。
- 证明：相同设置请求重复提交后版本保持不变；两个 Store 并发修改不同字段仍能原子合并且版本增加两次。

### 15. Agent 后端仍使用即将移除的工厂接口

- 模块：Runtime
- 主阶段：P6 装配 Agent 定义
- 相邻阶段：P5 执行准备、P10 工具执行
- 状态所有者：每次 Run 的 `RuntimeContext`
- 旧路径：向 Deep Agents 传入可调用的后端工厂；该接口已经弃用，并将在 Deep Agents 0.7 移除。
- 风险：升级依赖后 Agent 无法编译；若直接把某一次 Run 的后端实例写入缓存定义，又会让其他工作区错误复用它。
- 修复：按[官方后端协议](https://docs.langchain.com/oss/python/deepagents/backends)传入一个轻量代理实例，由它在每次工具调用时从 LangGraph 当前运行上下文取得本次 Run 的真实后端。
- 证明：定义缓存测试确认传入对象实现官方后端协议；真实模型工具循环和带审批的文件写入测试均通过，弃用警告消失。

### 16. 执行准备异常只记录日志，不结算 Run

- 模块：Runtime
- 主阶段：P5 执行准备
- 相邻阶段：P11 执行结算、P12 结果提交
- 状态所有者：当前租约拥有者和 `LocalStore`
- 旧路径：执行入口最外层捕获普通异常后只写日志；Run 保持 `running`、作业保持租约，最终被租约回收误判为资源清理无法确认。
- 风险：模型调用前的数据库或配置读取异常也会进入 `cleanup_required`，用户无法直接重试，并且错误事件缺失。
- 修复：只要没有出现资源清理失败，当前租约拥有者就重新绑定同一租约，把异常通过既有结算和原子结果提交路径写成 `run.failed`；租约已丢失或清理未确认时仍保持隔离策略。
- 证明：新增执行准备读取首次异常测试，Run 原子进入失败、作业进入死亡终态并产生结构化 `run.failed`；Run 作业测试全部通过。

### 17. 模型调用账本静默接受非法状态更新

- 模块：Runtime
- 主阶段：P8 模型调用
- 相邻阶段：P11 执行结算、P12 结果提交
- 状态所有者：`local_model_calls`
- 旧路径：记录首个输出和记录失败时不检查 SQL 实际更新行数；编号不存在或调用已经终结也被当作成功。
- 风险：模型输出已经交给 Agent，但持久账本缺失或状态冲突，最终用量与执行证明可能不完整。
- 修复：与既有成功结算规则一致，两种状态更新都必须命中一个仍处于保留或流式状态的调用，否则立即报错并由 Run 错误收口处理。
- 证明：新增不存在编号和终态后失败更新测试；模型账本测试全部通过。

### 18. 执行阶段保留了已经废弃的自动模型写回

- 模块：Runtime
- 主阶段：P5 执行准备
- 相邻阶段：P3.4 接纳检查、P8 模型调用
- 状态所有者：Run 接纳时冻结的具体模型
- 旧路径：执行阶段把已经持久化的模型再次写回，且吞掉写入错误；该路径只服务于已删除的自动模型选择。

### 19. 作业终态写回失败会泄漏工作槽

- 模块：Runtime
- 主阶段：P12 结果提交
- 相邻阶段：P5 作业领取、P11 执行结算
- 状态所有者：Runtime 进程内工作槽与持久作业租约
- 旧路径：`finally` 先查询 Run 并写回作业终态，之后才清理内存映射和释放并发槽；数据库异常会跳过后半段。
- 风险：一次收尾写入失败就永久减少一个并发槽，重复发生后 Runtime 不再领取任何任务；内存中的 Run 状态也无法回收。
- 修复：持久作业收尾允许记录错误并交给租约回收，进程内清理与工作槽释放放入不可跳过的内层 `finally`。
