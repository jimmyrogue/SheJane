# 代码质量审查记录

本记录按 Desktop App、Runtime SDK、Runtime 三个模块持续更新。目标不是整理格式，而是消除会造成数据损坏、重复执行、错误误判和恢复失败的问题。

## 审查方法

- 从公开入口向状态所有者追踪，不以单个报错位置为边界。
- SQL 重点检查事务原子性、外键、唯一约束、并发写入、迁移和失败回滚。
- 命令重点检查不可变载荷、稳定命令编号、重复投递和冲突响应。
- 错误重点检查分类是否跨 Runtime、SDK 和 App 保留，而不是只剩错误文案。
- 每项修复先增加能复现问题的测试，再执行模块测试；最终统一执行全量测试和 E2E。

## 已确认并修复

### 1. SQLite 外键只声明、未执行

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P2 幂等命令、P4 快照、P10 等待决定、P12 结果提交
- 状态所有者：`LocalStore`
- 旧路径：主连接和各个临时事务连接只配置行类型与忙等待，没有逐连接开启 `foreign_keys`。
- 风险：SQLite 默认关闭外键检查，孤立事件、作业、线程项或工具记录可能成功写入，后续只能表现为难以恢复的投影错误。
- 修复：所有 Runtime SQLite 连接统一执行相同连接配置并开启外键检查。
- 证明：新增测试分别从主连接和独立事务连接写入孤立事件，均必须触发 `IntegrityError`；相关作业和结果提交测试通过。

### 2. SSE 握手错误丢失结构化信息

- 模块：Runtime SDK
- 主阶段：P4 客户端读取快照并订阅变化
- 相邻阶段：P3.6 命令回执、P12 客户端收敛终态
- 状态所有者：Runtime HTTP 响应；SDK 负责无损翻译
- 旧路径：普通 HTTP 接口抛出带 `status/code` 的 `RuntimeHTTPError`，SSE 接口却重新抛出普通 `Error`。
- 风险：App 无法区分鉴权失败、Run 不存在、游标冲突和服务端错误，只能依赖易变化的错误文案。
- 修复：SSE 的非游标错误统一保留 HTTP 状态和 Runtime 错误码；游标重置仍使用专用错误类型。
- 证明：新增 SDK 测试验证 404、`run_not_found` 和原始消息均被保留。

### 3. 定时任务领取不是原子操作

- 模块：Runtime
- 主阶段：P3.5 原子提交
- 相邻阶段：P5 作业领取、P12 结果提交
- 状态所有者：`LocalStore`
- 旧路径：先查询到期任务，再更新状态，最后逐条重新读取；查询与更新之间没有数据库锁。
- 风险：两个 Runtime 或恢复进程可能同时拿到同一任务并各自启动 Run。稳定命令编号能防止部分重复，但不能替代任务领取本身的所有权。
- 修复：在独立连接的 `BEGIN IMMEDIATE` 事务中完成查询和状态更新，提交后只返回本事务领取的记录。
- 证明：新增两个 `LocalStore` 并发领取同一任务的测试，合并结果只能出现一次任务编号；定时任务测试全部通过。

### 4. 永久拒绝的待发送命令被无限重试

- 模块：Desktop App、Runtime SDK
- 主阶段：P2 客户端提交幂等命令
- 相邻阶段：P3 命令接纳、P4 客户端投影
- 状态所有者：Desktop 只拥有尚未被 Runtime 接纳的本地命令；Runtime 接纳后立即成为权威状态。
- 旧路径：SDK 吞掉每条命令的错误并只返回成功数量；App 只要数量不足，就每两秒重试整个未完成集合。
- 风险：工作区撤销、命令冲突、校验失败等永久 4xx 会产生无限请求；本地助手消息一直显示等待，用户也看不到真正原因。
- 修复：SDK 返回逐条失败及可重试分类；App 只自动重试网络、超时、限流和 5xx。永久拒绝会原子删除待发送命令并把本地助手投影结算为失败，同时保留其他线程继续投递。
- 证明：SDK 测试覆盖 409 与另一线程并行成功；IndexedDB 测试验证命令删除和失败投影在同一事务完成；App 相关测试与生产构建通过。

### 5. 请求校验错误被降级为通用 422

- 模块：Runtime SDK
- 主阶段：P3.2 请求校验
- 相邻阶段：P2 命令提交、P3.6 命令回执
- 状态所有者：Runtime 负责安全清洗校验错误；SDK 负责无损显示清洗后的结果。
- 旧路径：Runtime 返回 FastAPI 的安全错误数组，SDK 只识别字符串或对象，因此丢弃字段位置和消息。
- 风险：用户只看到 `Local Host HTTP 422`，无法知道哪个设置错误；上层只能再次自行解析响应，形成重复错误封装。
- 修复：SDK 识别清洗后的第一条字段错误，组合字段位置和消息；仍不读取或显示被 Runtime 删除的 `input` 和 `ctx`。
- 证明：SDK 测试验证状态码和 `body.memory` 错误消息均被保留。

### 6. 模型编号的客户端与服务端校验过宽

- 模块：Desktop App、Runtime SDK、Runtime
- 主阶段：P3.2 请求校验
- 相邻阶段：P2 模型选择、P5 执行准备
- 状态所有者：Runtime API 模型；SDK 对本地缓存值做同等的提前校验。
- 旧路径：`local:<供应商>:<模型>` 只检查冒号位置，供应商和模型中包含空白仍会通过，直到执行准备阶段才以“模型不存在”失败。
- 风险：损坏的本地缓存或手工输入会被当作有效选择，并生成本可在信任边界直接拒绝的持久命令。
- 修复：Runtime 和 SDK 统一要求供应商与模型部分都不含空白，同时保留模型编号内部的冒号。
- 证明：SDK 与 Runtime HTTP 测试均覆盖空白编号；通过 `sync-schemas` 重新生成 OpenAPI，线上的两个请求字段约束已同步，TypeScript 构建检查通过。

### 7. Skill 与 MCP 配置写入可能损坏或互相争用

- 模块：Runtime
- 主阶段：P6 绑定资源并取得 Agent 定义
- 相邻阶段：P3 配置请求接纳、P8 工具可见性
- 状态所有者：Runtime 管理的本地 Skill 与 MCP 配置文件。
- 旧路径：Skill 直接覆盖 `SKILL.md`；MCP 虽然先写临时文件，但所有请求共用同一个 `.tmp` 名称。
- 风险：写入中断会留下半个 Skill 文件；并发 MCP 请求可能互相覆盖临时文件或使其中一个替换失败。
- 修复：两者复用唯一临时文件、刷新文件内容后执行操作系统原子替换；失败时清理临时文件并保留旧文件。
- 证明：新增替换失败测试，确认旧内容不变且临时文件被清理；Skill 与 MCP 的 59 项测试全部通过。

### 8. SSE 合法 JSON 可以绕过事件协议

- 模块：Runtime SDK
- 主阶段：P4 客户端读取快照并订阅变化
- 相邻阶段：P8 事件翻译、P12 客户端投影
- 状态所有者：Runtime 生成事件；SDK 验证公开事件信封。
- 旧路径：SDK 只执行 `JSON.parse` 后强制转换类型，不验证 `event_type`。
- 风险：Runtime 字段漂移或代理返回错误 JSON 时，App 的事件分支全部无声跳过，Run 看似一直运行且没有明确协议错误。
- 修复：SDK 在 SSE 边界要求非空字符串 `event_type`，不合法信封进入既有的 `Malformed Agent SSE` 错误路径。
- 证明：新增合法 JSON 但缺少事件类型的测试；SDK 测试与构建通过。
