# Agent 运行时待优化记录

> 更新日期：2026-07-13
> 阶段编号以 [`harness-runtime-stages.md`](harness-runtime-stages.md) 为准。本文只记录目标取舍和待完成工作，不表示代码已经迁移。

## 阅读方式

每个阶段只回答四个问题：

1. 这一阶段是否需要保留？
2. 当前实现最主要的问题是什么？
3. 哪些旧机制应删除或替换？
4. 完成迁移后如何验证？

优先级：

- **A：基础正确性**。不先完成，后续迁移会继续制造双状态或恢复漏洞。
- **B：运行时独立**。让桌面端使用自带密钥（BYOK）时可以脱离 SheJane Cloud 工作。
- **C：执行质量**。在状态边界稳定后改进上下文、验证、工具和记忆。

## 总体结论

### 必须保留

- 稳定的客户端与运行时协议。
- 幂等命令和权威任务状态。
- 可恢复事件订阅。
- 持久作业、租约和执行尝试。
- LangGraph 检查点。
- 模型调用账本和工具回执。
- 工具权限与人工确认。
- 执行结束时的资源清理。
- 最终结果的原子提交。

### 应优先删除

- 本地运行时失败时自动改走云端任务。
- Renderer 与运行时各自保存一套权威聊天历史。
- HTTP 请求处理函数直接创建后台 Agent 任务。
- 共享内存队列充当事件事实来源。
- 复制 LangGraph 私有检查点数据库行。
- 多个中间件同时控制模型循环去向。
- 按工具名称整批放行的权限缓存。
- 对产生副作用的工具做通用自动重试。
- 默认执行的反思模型调用。
- 每次任务自动把目标和最终回答写入长期记忆。
- 从 SSE 事件重新计算用量，或扫描最后一条 AI 消息寻找最终回答。
- 客户端根据连接结束或已见事件猜测任务终态。

### 当前明确不做

- 不在运行内核中内置飞书连接器、消息同步和待办提取。
- 不引入 Kafka、Redis、Celery 或 Temporal 作为第一版运行时依赖。
- 不建立只有一个实现的通用框架、工厂或策略语言。
- 不承诺外部副作用“恰好执行一次”。
- 不实现多模型供应商自动回退。
- 不开发移动端 Harness。
- 不在状态边界稳定前增加复杂向量记忆或模型评审链。

## P1：桌面端连接运行时

**必要性：保留。** 需要删除的是“Electron 必须启动 Python”这个假设。

改造前问题：

- Electron 先选择端口再启动 daemon，存在端口被抢占的窗口。
- 健康检查同时承担存活、就绪和身份确认。
- Renderer 可以接触原始连接信息和配对凭证。
- daemon 启动失败后仍存在云端回退路径。
- 关闭进程时缺少完整的有限等待和资源排空。

目标调整：

- 只保留“桌面托管本地运行时”和“外部运行时”两种所有权。
- 运行时自行绑定系统分配的端点，再通过启动通道报告地址。
- 分开存活检查、就绪检查和协议初始化。
- Electron Main 保管凭证，Renderer 只使用会话句柄。
- 连接失败时保留用户输入并提供重启入口，不创建云端任务。

完成标准：

- 本地、外部本机和远程运行时使用同一初始化协议。
- 桌面端不会停止自己不拥有的运行时。
- 启动超时、异常退出和版本不兼容都有明确状态。

优先级：**B**。

当前进度：**桌面端凭据边界已完成。** Electron Main 独占本地 Runtime 的原始配对 Token，并通过 Electron 请求会话只向目标 Runtime 地址注入认证头；preload 只向 Renderer 暴露 Runtime 地址和不含秘密的桌面会话标记。浏览器构建中的 Token 回退已删除。P1 其余工作仍包括由 Runtime 自行报告端点、独立就绪握手和托管进程的有限关闭。

## P2：客户端提交命令

**必要性：保留。** 客户端需要可靠投递，但不应拥有任务状态机。

当前问题：

- 已完成第一批投递基础：桌面客户端为本地 Run 生成稳定命令编号和客户端消息编号，网络异常时使用原内容重试一次。
- 已完成第一批投递基础：运行时持久保存不可变创建命令；同编号同内容返回原 Run，同编号不同内容返回冲突。
- 已完成对话所有权迁移：助手占位只是 `pending` 临时投影；已有 Runtime 对话不再上传客户端历史，旧本地对话只允许第一次迁入时携带一次历史快照。
- 取消、问题回答、权限决定、计划审批和工具对账已迁入统一命令协议；重试和编辑复用 `run.start`，检查点分叉使用 `run.fork`，都进入同一待发队列。可变回退不再单独实现。
- 已完成持久投递：客户端重启后会扫描并重发待确认命令，收到运行时回执后才删除。

目标调整：

- 客户端只保存待发送命令和可丢弃的界面投影。
- 每个命令带稳定编号；重复发送必须得到同一回执。
- 创建、取消、恢复、审批、编辑和分支都使用同一命令结构。
- 对话正文、助手消息和任务状态由运行时保存。

完成标准：

- 网络在提交前、提交中和提交后断开时都不会重复创建任务。
- 清空客户端缓存后可以从运行时重建对话。
- 同一资源的冲突编辑能通过版本前置条件被拒绝。

优先级：**A**。

当前进度：**P2 已完成。** 本批主阶段是 P2，上游契约是 P1 的已认证运行时会话，下游契约是 P3 的幂等接纳事务。待确认命令由客户端 IndexedDB 队列拥有，Runtime 的 `local_commands` 保存权威命令与稳定回执，替代了检查点分叉由界面直接调用旧接口的路径。创建、重试、编辑、检查点分叉、取消和四类等待决定共用同一待发队列、稳定命令编号和按线程投递顺序，收到 Runtime 回执后才删除。重试与编辑复用 `run.start`；编辑以仍未被替换的原用户消息作为冲突前置条件并保留旧记录。检查点分叉使用 `run.fork` 创建新的产品对话和明确的图分支头，并持久保存协议版本和所需能力。快速重复点击由执行锁合并；传输失败或重启后再次点击同一来源检查点时，会复用并导航到已有待发分支，不生成第二个命令。网络中断或客户端重启后按原内容重投，传输失败期间只保留待确认投影，不伪造失败终态。可变回退会改写历史且与不可变分支重复，因此不提供单独命令；用户需要回到旧位置时创建分支。后台接纳后由快照和变化游标同步权威状态。相同命令重复发送返回原回执，相同编号改变内容返回冲突。界面临时投影与创建或分叉命令在同一客户端事务保存；删除投递中的对话时，接纳后会先取消 Run、等待权威终态，再删除 Runtime 线程。已有 Runtime 对话的历史由 Runtime 消息生成，客户端历史只用于一次旧数据迁入。

## P3：运行时接收命令

**必要性：保留，但合并成一个清晰的接收事务。** 不建立额外的接纳服务或幂等服务。

### P3.1 身份认证

当前 loopback + Bearer Token 只适合本机配对。目标是将凭证映射成运行时身份，再由 P3.4 做资源授权。远程连接必须单独使用安全传输、设备身份、过期和撤销机制。

完成标准：伪造客户端信息不能获得身份；过期或撤销凭证失效；模型密钥与客户端凭证完全分离。

当前进度：本机配对 Token 已映射为稳定的 `local:owner` 身份；Run、定时任务、作业输入和命令日志都保存该身份，Token 轮换不会改变资源所有者。认证中间件已改为纯 ASGI，并只接受真实凭证，不信任客户端声明的身份。身份必须由入口显式传入；执行作业时还会与所属 Run 的身份核对，缺失或冲突会安全失败，不会自动补成本机所有者。旧数据库中的 Run、命令、定时任务和待执行作业会迁移到本机所有者。远程设备身份、凭证过期与撤销仍未实现，资源读取和修改权限由 P3.4 继续收口。

### P3.2 请求校验

用一个稳定的命令联合类型替换松散字典。OpenAPI 生成客户端类型，未知字段在稳定接口上直接拒绝。请求大小、列表长度、嵌套深度和标识格式必须有上限。

完成标准：Python 模型、OpenAPI 和 TypeScript 类型一致；无效请求不写数据库。

当前进度：创建任务已拒绝未知字段，并限制目标、历史条数、路径、模型编号、设置与元数据的嵌套深度和节点数量，以及整体持久化载荷大小；所有本地 HTTP 请求体还会在 JSON 解析前执行 1 MiB 硬限制。取消和四类等待决定使用严格的命令联合类型；创建、重试和编辑使用严格的 `run.start` 请求，检查点分叉使用严格的 `run.fork` 请求。OpenAPI 与客户端类型保持生成一致。可变回退不属于公开命令。

### P3.3 幂等

增加持久命令日志，以“身份 + 命令编号”为唯一键。相同内容返回原回执，不同内容返回冲突。数据库唯一约束负责关闭并发窗口。

完成标准：两个并发相同请求只产生一个任务；daemon 重启后仍能返回原回执。

当前进度：所有现有运行命令都要求稳定命令编号并写入 `local_commands`。创建、重试、编辑和检查点分叉在同一事务创建 Run、作业、消息和稳定回执；取消与四类等待决定也在各自接纳事务保存稳定回执。同内容重放返回原回执，不同内容返回冲突。命令唯一约束是“身份 + 命令编号”，同一个 Run 可以接收多个不同命令；旧数据库命令会自动迁移。

### P3.4 接纳检查

在写入前确认工作区、对话、模型配置、凭证引用、能力和版本条件。只做本地确定性检查，不连接模型或 MCP。

完成标准：无权限资源和无效模型配置不会先创建任务再失败。

当前进度：资源授权、协议前置条件和设置快照基础已经完成。工作区、Run 和定时任务都按 Runtime 身份隔离；读取 Run、流、诊断、产物，以及取消、恢复、分支、追加指令、权限审批、问题回答和计划审批，都会先确认所属 Run。创建任务会在命令接纳事务中确认工作区和父任务归当前身份，工作区必须真实存在；命令重放仍先返回原回执，不会因授权后来撤销而破坏幂等。工作进程每次开始或恢复前会再次核对 Run 的工作区，定时任务和分支也会拒绝已经撤销的工作区。模型不能再通过 `workspace.open` 自行扩大文件范围。

新任务和检查点分叉必须声明协议版本和所需能力；Runtime 提供经过认证的能力查询接口，并在写入前拒绝不兼容请求。普通任务和定时任务都会在接纳时把有效运行设置冻结为版本化快照，后续环境默认值变化不会悄悄改变已接纳任务；未知快照版本会安全失败。命令日志、Run 和定时任务只保存公开设置字段与受限工作流元数据。

模型接纳已与第一批 P6 供应商绑定连通。对于 `local:<供应商编号>:<模型编号>`，接纳事务会确认供应商属于当前身份、处于启用状态、模型真实存在于本地目录，并明确声明支持流式输出与工具调用；执行前会再次核对供应商版本和凭据，配置更新或密钥撤销后，尚未开始的旧作业会安全失败。真实密钥只存在操作系统凭据库，数据库和 Run 快照只保存引用。上下文窗口现在作为显式能力资料保存，但尚未用于统一上下文预算；云端中转的具体模型目录仍由 Go 服务负责。因此 P3.4 对本地直连模型已经完成主要检查，对旧云端中转仍保留现有边界。

### P3.5-P3.6 原子提交与回执

同一 SQLite 事务写命令日志、用户消息、任务、待执行作业和资源变化。HTTP 处理函数不创建异步 Agent 任务，也不创建进程内事件队列。

完成标准：在每条数据库写入之间模拟崩溃，不会出现“有任务没作业”或“有作业没任务”。回执明确表示“已提交，尚未领取”。

当前进度：创建、重试、编辑和检查点分叉命令会在同一 SQLite 事务提交命令日志、对话、用户消息、助手消息、Run、`pending` 作业和线程变化。取消命令、Run 的取消请求和命令回执也由单个事务提交；问题回答、权限决定、计划审批和工具对账命令在一个事务内写决定、事件和回执，等待周期完整时同时写恢复作业。工具对账还在同一事务结算当前及祖先任务中结果不明的工具回执。事务完成后协调器只负责唤醒或停止执行任务。已有 Runtime 对话的上下文由 Runtime 从权威消息生成，不再信任客户端上传的历史；旧版客户端对话只在第一次进入 Runtime 时导入一次。

优先级：**A**。

## P4：快照和可恢复订阅

**必要性：保留。** 共享队列和 `[DONE]` 不是可靠协议。

当前问题：

- SSE 没有可靠的恢复游标。
- `MAX(seq)+1` 缺少并发唯一约束。
- 逐字事件和状态变化都走同一种持久化路径。

目标调整：

- 客户端先读快照和高水位，再回放后续事件。
- 每个订阅者维护自己的游标并从数据库读取。
- 通知中心只发送可丢失的唤醒信号。
- 状态变化持久化；逐字输出和推理进度仅临时传输。
- 增加全局资源变化游标，让多个客户端发现其他客户端创建的任务。

完成标准：

- 在快照、回放、注册和实时通知的每个位置断线都能恢复。
- 慢客户端不会让其他客户端丢事件。
- 游标过期会要求完整重同步，而不是从错误位置继续。
- SSE 结束不会改变任务状态。

当前进度：**P4 已完成。** 每个订阅者从数据库事件日志维护独立持久游标；Runtime 会拒绝超出事件窗口的游标，客户端收到统一重同步信号后丢弃本地对话投影、读取权威线程快照，再从可读位置继续订阅。助手消息投影原子记录正文已覆盖的事件序号，因此等待、终态、旧库升级和运行中恢复都不会重复正文。逐字文本、推理、临时用量、工具参数片段和子 Agent 启动片段不再写入 `local_events`，而是通过每个订阅者独立的有界实时队列发送；断线或背压时允许丢失，最终界面由助手草稿、结果事务和线程快照纠正。升级时会删除旧版本遗留的这五类临时事件，避免首次重连重复正文；原事件序号允许保留空洞。工具请求、工具结果、权限、等待、错误和终态仍持久化并可重放。每个 Run 使用独立的发布锁，注册订阅者和读取初始事件位于同一短临界区；初始化失败也会移除订阅者。实时队列中的持久事件只作为唤醒信号，客户端始终按数据库游标补齐，因此跨进程提交、队列背压或较早通知缺失都不会跳过较小序号。当前不主动裁剪事件，因此不增加保留任务。

优先级：**A**。

## P5：持久作业和执行监督

**必要性：保留。** 内存中的 `asyncio.Task` 不能承担跨重启可靠性。

当前问题：

- HTTP 已只提交持久作业，Runtime 工作进程负责领取和执行。
- 并发名额在领取作业之前取得；租约保存所有者、代次、期限和尝试次数。
- 运行期间持续续租；取消先持久化，等待用户时结束当前尝试，恢复命令创建新作业。
- 尚未领取的作业可跨 Runtime 重启继续执行；已经开始且结果不明的尝试不会自动重跑。
- 执行上下文只在当前尝试内存在；权威设置、消息、检查点、回执和结果保存在 Runtime。

目标调整：

- 使用 SQLite 作业表保存 `pending/leased/completed/dead/canceled`。
- 工作进程先取得执行名额，再领取作业。
- 所有执行期写入都校验租约所有者和代次。
- 只有能证明尚未产生输出或副作用的暂时失败，才允许通过新作业重排；当前不启用自动模型重试。
- 等待用户时结束当前执行尝试；恢复命令创建新作业。
- 关停时只重排已经确认静止的执行尝试。

完成标准：

- daemon 在领取前和领取后崩溃都能恢复。
- 旧代次不能写事件、检查点、工具回执或终态。
- 并发数严格受限，没有已领取作业在进程内排队。
- 取消完成意味着模型、MCP 和子进程都已经停止。

优先级：**A**。

当前进度：**P5 已完成。** 命令接纳事务创建 `pending` 作业；工作进程先取得本机并发名额，再原子领取作业并写入租约所有者、代次、期限和尝试次数。同一 Run 由数据库唯一约束保证最多一个活动作业；旧所有者或旧代次不能续租、提交结果、移动检查点或更新工具与模型账本。等待用户会先保存检查点并结束当前作业，决定命令再创建新的恢复作业。

取消请求先进入数据库，再取消执行协程；模型、MCP、文件后端和子进程退出统一资源作用域后，Runtime 才提交取消或失败结果。关停会停止领取、取消活动尝试并有限等待同一清理路径；无法确认资源静止时拒绝完成关停。租约过期或清理失败会把原作业和 Run 隔离为 `cleanup_required`，记录模型与工具“结果不明”，且不释放给新代次；只有原所有者提交清理证明后才能结算失败。

这里明确不增加“租约过期自动重领”。外部动作可能已经成功，盲目重跑会重复副作用；当前安全策略是隔离、核对或由用户发出新的显式命令。尚未领取的 `pending` 作业仍可在 Runtime 重启后继续领取，这已经覆盖可证明安全的崩溃位置。P10 的持久工具回执、P11 的统一资源清理和 P12 的租约保护结果事务已经补齐原文所列前置条件。

## P6：资源绑定和 Agent 定义

**必要性：保留，但删除每个任务重复编译整张图的做法。**

当前问题：

目标调整：

- 用结构指纹复用不可变 Agent 定义。
- 模型连接、MCP、子进程和密钥只属于本次执行，并在 P11 关闭。
- BYOK、本地模型和 SheJane 中转服务使用同一供应商绑定接口。
- 业务平台集成只通过标准工具或 MCP 绑定，不再增加平台专用接口、数据表和界面状态机。
- 动态提示内容移到 P8，不放进可复用定义。

完成标准：相同结构重复任务不重新编译；不同任务不会串用工作区、密钥、权限或事件出口；关闭 SheJane Cloud 后仍能完成模型→工具→模型循环。

优先级：**B**。

当前进度：**第一批供应商绑定基础已经落地。** Runtime 现在拥有按身份隔离的供应商目录、操作系统凭据库引用和本地模型目录；首个适配器支持 OpenAI 兼容接口，包括 Ollama 等无需密钥的本地服务。桌面设置页可添加、修改和删除供应商，模型选择器会合并本地目录与可选云端目录；没有云端中转时不再展示不可执行的自动模式。桌面客户端在云账号恢复失败后仍可直接进入本地 Runtime，发送只走 daemon；daemon 不可用时明确失败，不会转发到云端。已经通过“模型调用→本地工具→模型继续回答”的直连测试，且没有自动回退到云端。

供应商更新和删除与任务准入、执行前复核共用同一把按身份和供应商隔离的锁。变更会先取消仍绑定旧版本的活动任务，并等待进程内执行确认停止；不能按时停止就拒绝变更。密钥使用不可变的版本化凭据引用：先创建新凭据，再由 SQLite 原子切换引用，失败不会覆盖旧配置使用的密钥；提交后才清理旧条目。从“需要密钥”切换为“无需密钥”或删除供应商也会清理旧凭据。直连模型异常在进入日志、事件和 SSE 前会按本次密钥强制脱敏。每次 Run 还会冻结云工具是否真实可用；没有云会话时不装配 `web.search`、图片、云端 PDF 和云端代码执行工具，只保留本地工具，避免把不可执行能力展示给模型。

桌面会话统一归属稳定的本地 Runtime 身份，不再按可选云账号切换数据库；旧匿名会话和旧账号会话会一次性合并到本地库，同一会话保留更新时间较新的版本。云端能力发现未完成或明确不可用时，客户端都不展示自动模式，并拒绝发送当前目录中不存在的模型，避免发现失败时误放行。

第二批资源作用域基础已经开始：每个被领取的执行尝试现在拥有一个标准库 `AsyncExitStack`，本地模型密钥只作为该协程的局部变量传递，不再保存在协调器共享字典中；本次创建的 OpenAI 兼容模型客户端会在完成、等待、失败或取消离开作用域时关闭。当前 MCP 适配器保持官方默认的无状态模式，每次工具调用自行创建并清理会话，因此不增加常驻 MCP 连接池。

第三批运行时上下文迁移已经完成：LangGraph 现在声明明确的 `RuntimeContext`，P7 调用时注入任务、身份、工作区、主模型连接、工作区后端、记忆开关、本地存储和事件出口。动态任务提示由模型调用中间件读取上下文后生成，不再固化在系统提示闭包中；运行中追加指令、进度记录、记忆检索、记忆回写和云端代码工具也从上下文取得任务或工作区。工作区后端使用 Deep Agents 原生后端工厂，同一检查点适配器通过当前异步任务的租约上下文完成代次保护，不再绑定某一次执行。

第四批 Agent 定义复用已经完成：Deep Agents 内建摘要和子 Agent 使用任务局部模型代理，模型连接和密钥只存在于本次执行；MCP 工具在可复用图中只保留名称、说明和参数结构，真实工具对象、服务器配置和请求头通过任务局部代理绑定。MCP 发现会逐页读取并限制候选数量、时间、HTTP 响应字节、说明和参数结构大小；工具名称、说明和参数结构还会检查已配置凭据，拒绝内容不会原样写入日志。尚无等价内存边界的 WebSocket MCP 暂不执行。模型型工具筛选和工具结果评审因隐藏模型调用且与确定性预算、P9 验证重复，已连同配置入口删除。Runtime 按模型能力资料、工具结构、子 Agent 定义、技能与记忆来源和中间件参数生成结构指纹；相同结构复用进程内有界 Agent 定义，不同结构重新编译。当前 P6 的目标边界已经完成；未来若引入常驻 MCP 会话或子进程，仍必须注册到 P11 的执行资源栈。

文件资源边界也已收口：有授权工作区时，文件工具只能写入该工作区；没有工作区时，每个执行尝试使用独立的临时目录，并由 P11 资源栈删除，不再共用 `~/.shejane/workspace`。临时目录删除失败会进入既有的清理失败隔离路径，不能伪装成清理完成。Skills 和 `AGENTS.md` 来源通过只读后端挂载；单文件来源不会顺带暴露同目录的其他文件，中间件仍可加载内容，但模型文件工具不能修改或上传文件到这些来源。主要阶段为 P6，上游由 P5 交付冻结的工作区和执行代次，下游 P7 只收到已绑定后端；被替换的旧路径是共享临时目录和可写的指令来源挂载。

## P7：LangGraph 调用、恢复和分支

**必要性：保留。** 这是图执行与运行时状态之间的明确边界。

当前问题：

- 产品任务编号直接当作 LangGraph 线程编号。
- 手工读取最终快照判断完成或等待。
- 分支通过复制 LangGraph 私有 SQLite 表实现。
- 事件仍使用旧的三元流翻译路径。

目标调整：

- 分开产品任务编号、图线程编号和检查点编号。
- 新任务、恢复、中断继续、分支和重放使用明确输入类型。
- 使用 LangGraph 公开的检查点和时间旅行接口。
- 使用同步持久化，确保进入下一步前检查点已经提交。
- 使用统一事件接口，产品事件与原始图诊断分层。

完成标准：恢复不会重复已提交的图步骤；分支不复制私有表；不同分支都有明确的检查点头。

优先级：**A**。

当前进度：**P7 的当前产品边界已经完成。** 产品任务编号、LangGraph 线程编号和检查点分支头已分开保存；新任务拥有独立图线程，checkpoint 分支共享原图线程但拥有各自的明确分支头。分支通过公开的 `thread_id + checkpoint_id` 配置启动，不再读取或复制 LangGraph 私有 SQLite 表；选择的 checkpoint 必须位于来源任务当前分支的祖先链上，不能跳入兄弟分支。Agent 定义指纹包含明确的持久状态兼容版本，不会因为 LangGraph 或 Deep Agents 的兼容补丁升级而永久阻断恢复；真正改变状态语义时必须提升该版本。

执行流已经切换到 LangGraph v2 类型化事件，使用同步持久化，并从 `checkpoints` 流逐步更新租约保护的分支头；恢复始终从该明确头部继续。每个 checkpoint 还保存产品任务和执行尝试归属，形成可恢复日志；如果进程恰好在 checkpoint 提交后、产品分支头更新前退出，启动、恢复、再次执行或分支前会沿公开父链单调对账，不会从旧头重复已提交步骤。v2 的 checkpoint 事件早于待处理中断写入，因此完成和等待判断仍通过公开的 `aget_state()` 精确读取当前分支头；实验性的 v3 生命周期流稳定后再评估替换，不在核心路径提前采用。原始节点状态不再进入产品 SSE，只保留稳定的模型、工具、人工确认和自定义事件，原始图状态由 checkpoint 与 tracing 诊断层负责。纯重放暂不开放：在 P10 的副作用工具回执和结果不明核对完成前，公开重放会有重复执行副作用的风险。

## P8：模型回合和上下文

**必要性：保留。** 但应只有一个上下文预算流程和一个供应商边界。

当前问题：

- 历史截断、Deep Agents 摘要和 ContextEditing 各自处理上下文。
- 通用 PII 中间件会直接改写模型输入。
- 模型型工具筛选器增加隐藏调用和不稳定性。
- 用量通过 SSE 事件累加。
- Auto 模型解析和云端回退仍依赖 Go API。

目标调整：

- 按“删噪声→转存大结果→取回必要内容→最后摘要”的顺序处理上下文。
- 敏感信息按工作区和供应商策略处理；密钥永不进入模型输入。
- 工具可见性使用确定性规则，不额外调用筛选模型。
- 模型调用前预留预算，结束后在持久账本中结算或标记结果不明。
- 自动重试只允许发生在尚未输出内容、明确暂时错误、同一模型且预算足够时。

完成标准：断线和重复事件不改变用量；模型产生内容后不会从头拼接另一次重试；没有任何静默供应商回退。

当前进度：**P8 已完成。** daemon 已删除按消息数二次截断、关键词拼接的早期摘要和额外 `ContextEditingMiddleware`，上下文压缩统一交给 Deep Agents 内建的文件转存与令牌感知摘要。旧本地对话首次迁入 Runtime 时，client 只保留与 1 MiB 接口上限对齐的传输边界，不再伪造语义摘要。

身份与安全提示的所有权已经从 Go 中转迁回 Runtime。`identity.md` 作为最高优先级、不可截断的结构层，位于 Deep Agents 基础提示和开发者指令之前，因此本地模型、BYOK 和可选中转获得相同边界；内建及用户配置的子 Agent 也在自己的系统提示开头复用同一基线。Runtime 请求用 `prompt_owner=runtime-v1` 声明提示已装配，Go `/agent/llm/stream` 对这类请求不再追加第二份 `agent_local` 提示，只转发消息、调用模型并处理计费；尚未删除的旧 Web 调用和旧 Runtime 没有该标记，暂时保留兼容提示，避免滚动升级时失去原有边界。主要阶段为 P8，上游 P7 提供只读执行上下文，下游 P9/P10 仍由确定性完成路由和工具策略执行真实权限；被替换的旧路径是“只有走 SheJane Cloud 才有身份与安全基线”。

通用 `PIIMiddleware` 和客户端可填写的脱敏开关已删除：它会直接改写 LangGraph 状态，电子邮件和网址也并不天然是不可出站的秘密。兼容的 `SHEJANE_LOCAL_PII_REDACT` 部署策略仍保留，但现在由 `OutboundPolicyMiddleware` 只处理发往外部供应商的请求副本，不改写检查点中的用户原文。已知模型凭据不论本地还是外部供应商都会在该边界强制删除。未来增加工作区级“仅本地/允许外发”时，继续复用这一个供应商边界，不再建第二套脱敏链。

通用 `ModelRetryMiddleware` 和可调模型重试次数也已删除。持久账本虽然已经能够区分调用是否产生可见输出，但当前仍选择明确失败，不自动重新执行整次模型调用；未来只有在错误明确属于瞬时故障、尚未输出、同一模型且预算充足时，才可重新评估是否增加有界重试。

模型调用账本已落地。`local_model_calls` 在联系供应商前原子预留本次调用和运行级次数预算；文本、推理或工具调用首次产生时，会在向客户端展示前持久化“已输出”。完整响应结束后记录供应商请求编号、输入/输出 token 和云端费用；供应商不返回用量时显式标记“已完成但未计量”，连接中断、取消或 Runtime 重启留下的未完成调用标记“结果不明”，不当成零用量。`run.completed` 从账本聚合权威用量；`llm.usage` SSE 只保留为可丢失的实时显示，断线和事件重放不会改变结算。

模型预算现在同时覆盖输入、输出、时间、调用次数和可计算费用。模型资料可声明输入/输出 token 上限；未知模型使用保守的部署默认值，OpenAI 兼容直连和可选中转都会把输出上限传到供应商。所有模型请求都有硬超时。中转调用继续由服务端预留和结算额度；BYOK 在没有价格资料时不虚构费用，只记录用量并通过次数、输出和时间限制资源。

账本已取代进程内 `ModelCallLimitMiddleware`，预算跨恢复和新执行尝试仍然生效。当前不恢复自动模型重试：安全的“输出前重试”还需要明确的临时错误分类和足够费用预算，不是 P8 完成的必要条件。

权威助手草稿已开始由 Runtime 保存。LangGraph 顶层模型节点完成一轮后，Runtime 才把组装完整的助手文本和工具调用写入 `local_assistant_drafts`；逐字增量仍只用于显示。同一模型消息重放不增加版本，后续模型回合按版本单调替换。任务完成时优先读取该草稿，只为兼容旧 checkpoint 保留从图状态提取最终文本的回退。P12 还需将最后草稿与任务结果原子提交为对话消息。

每次真实供应商调用还会执行确定性的上下文硬边界。边界优先读取模型资料中的 `max_input_tokens`，预留供应商封装和工具结构空间；超大单条消息保留头尾并加入明确截断标记，总历史仍超限时再保留系统指令和最近内容，加入“早期输入已省略”标记。该边界同时覆盖 Deep Agents 的摘要模型调用，避免 `trim_tokens_to_summarize=None` 在超大首次输入上再次溢出。如果可见工具结构本身已占用大部分窗口，运行时会在联系供应商前明确失败，不发送必然溢出的请求。

工具全集属于持久图定义，不能按当前一句自然语言增删，否则 checkpoint 分叉和省略式后续指令会改变图指纹或丢失能力。因此 Office 工具始终注册在稳定图中，只在每次模型请求的副本上确定性隐藏其大结构；判断会读取当前目标、保留的消息和历史工具调用。新目标需要 Office 时可立即恢复可见，上一轮操作过 Office 时“继续改第二页”也不会丢失工具。相同筛选还在最终供应商边界覆盖通用子 Agent、配置型子 Agent和摘要调用，避免它们绕过主 Agent 的请求处理中间件。该筛选不调用额外模型、不改变 checkpoint，也不改变图定义编号。

最后一个旧分支也已删除：可选 Go 中转不再在上游失败后自动选择下一候选并重跑。所选模型失败时会释放额度预留、记录失败并返回明确错误；改用其他模型只能来自新的显式命令。因此直连和中转都满足“单次运行只绑定一个模型、没有静默回退”的完成标准。

优先级：**B/C**。

## P9：输出路由和完成判断

**必要性：保留为一条确定性的 LangGraph 路由，不建立路由服务。**

当前问题：OutputGuard、VerificationLoop、ProgressLedgerGuard 和 PlanApproval 都可能在模型后控制跳转，规则互相竞争。

目标调整：

- 有工具调用就进入 P10；没有工具调用才形成最终候选。
- 只检查用户明确要求、工具后置条件或已接受计划中的验收项。
- 验证证据保存为持久回执。
- 可修复问题写入运行时状态后返回 P8，不伪装成用户消息。
- 预算耗尽时明确失败或受阻。

删除：拒答关键词正则、默认计划审批、进度守卫和默认模型评审。

完成标准：任一模型回合只有一个路由决定者；未满足明确验收项的任务不能被提交为成功。

当前进度：**P9 已完成。** `OutputGuardMiddleware`、`VerificationLoopMiddleware`、`ProgressLedgerGuardMiddleware` 和默认 `PlanApprovalMiddleware` 已从执行图删除。工具调用继续由 LangChain 内建工具条件进入 P10；没有工具调用时，只有 `CompletionRouterMiddleware` 检查完整消息、无效工具调用、输出截断、内容过滤和最新的 `task.verify` 结果。拒答关键词不再参与判断，`task.progress` 也不再是完成前强制动作。

验证失败时，修复要求写入持久图状态，并在下一次模型请求的系统上下文中以 `<runtime-repair>` 注入；不会伪装成用户消息，也不会把不可信的工具错误原文提升为系统指令。路由和修复计数绑定当前任务编号，checkpoint 分支不会继承来源任务的修复提示或预算。最新 `task.verify` 的 `ToolMessage`、工具调用编号和路由决定都随同步 checkpoint 保存，可作为当前阶段的持久验证回执；验证通过后若又执行副作用、未知或失败工具，旧回执立即过期并要求重新验证。修复次数耗尽、空输出、截断、过滤、混合有效与无效工具调用会变成结构化 `run.failed`，不会以空白或未验证答案提交成功。

`task.verify` 本身也已收紧到 Runtime 安全边界：文件检查必须位于当前工作区真实路径内，没有工作区时明确拒绝；网址检查复用 `web.fetch` 的出站策略，禁止带凭据网址、回环、私网、链路本地和保留地址，并逐跳检查重定向。旧计划审批表、接口和客户端显示只用于兼容已持久化历史，不再产生新的计划审批中断；P10 将统一清理人工等待结构。

优先级：**C**。

## P10：工具、权限和人工确认

**必要性：完整保留。** 模型提出的动作必须先经过不可信输入边界。

当前问题：

- 通过工具名称判断危险程度。
- “本次任务允许”只缓存工具名，不限制路径、域名、命令或次数。
- 通用工具重试可能重复产生副作用。
- 权限、用户问题和计划审批使用不同结构。
- 工具结果有多套失败解析和重试逻辑。

目标调整：

- 每次工具调用有稳定操作编号、参数指纹和持久回执。
- 权限策略根据具体参数、资源和风险决定允许、询问或拒绝。
- 可复用授权必须限制工具版本、资源范围、风险、有效期和次数。
- 整批工具先检查；任一调用需要询问时，执行前暂停整批。
- 权限、用户输入、计划审核和结果核对使用同一种等待请求结构。
- 只读工具可对明确暂时错误有限重试；有副作用工具必须具备幂等或核对协议。

完成标准：外部动作成功但本地未知时不会盲重试；编辑后的参数重新检查；大结果只以安全摘要和工作产物引用进入模型。

当前进度：**P10 已完成。** 旧的 `DESTRUCTIVE_TOOLS`、Deep Agents 默认人工确认中间件和协调器内“按工具名允许”缓存已经删除。`ToolReviewMiddleware` 会在执行前检查工具是否属于当前稳定图、参数是否符合工具结构，并为每个调用计算操作编号、参数指纹和风险等级。只要同一模型回合中有一项需要确认，整批都会在任何工具开始前暂停；批准、修改或拒绝必须先写入统一等待候选，再按原顺序恢复。任意未持久化的通用恢复请求会被拒绝。

`ToolExecutionMiddleware` 现在覆盖主 Agent 和所有子 Agent。每个调用都保存 `prepared / running / paused / completed / failed / rejected / canceled / outcome_unknown` 回执；完成结果可安全复用，同一模型批次内重复的调用编号会在生成回执前拒绝。操作编号同时包含检查点命名空间、模型批次、工具版本和参数指纹，因此供应商在后续回合重复使用调用编号也不会碰撞。只有明确只读且资源不冲突的调用可并行；存在写入或外部动作时，整批按模型原始顺序通过公平读写门执行，恢复时使用绝对批次位置而不是重新编号。云端付费工具继续使用 Run 与工具操作编号组成的幂等键，文件和其他本地副作用依靠工具回执阻止恢复时重复执行。

产生副作用后结果不明会生成显式核对等待，不会自动重跑。用户只能选择“确认已完成”“确认未执行并重试”或“终止”；核对会沿数据库中的真实父任务链查找祖先回执，逐层验证所有者，并在一个事务中更新祖先回执、当前回执和等待候选。这个核对不依赖客户端元数据，也不因图版本升级而失效。全部决定完成但进程尚未恢复的等待周期会在运行中和 daemon 重启后自动重新入队。

人工等待已经有统一的 `local_wait_candidates` 权威记录，当前承载工具确认、`user.ask` 和结果核对；一个等待周期可包含多个 LangGraph 中断，并按等待周期编号与中断编号精确恢复。旧计划审批不会再产生新记录，只保留历史兼容。工具确认和问题回答都支持相同决定的幂等重放，冲突的第二次决定返回错误。运行级授权只复用完全相同的工具、参数指纹、风险和图版本；有效期为 24 小时，最多复用 20 次，同一操作重复提交不会重复消耗次数。

LangGraph 只要仍有后续节点，就必须同时提供可持久化的中断；如果出现“有后续节点但没有中断”的不一致快照，Runtime 会明确失败，不再写入无法恢复的空等待状态。

工具结果进入模型前限制为 64 KiB；更大的结果保存为工作产物，只给模型安全预览与引用。单次工具产物最多保留 16 MiB，存储层另有单项、单任务、单所有者和全库配额。MCP 的 HTTP、SSE 和本地进程传输都在解析前限制单次响应大小；WebSocket 在具备同等边界前不装入 Agent。诊断接口只暴露等待状态和脱敏后的工具回执身份、状态与哈希，不返回原始工具结果。

优先级：**A/C**。

## P11：执行结算和资源清理

**必要性：保留，并覆盖所有结束方式。** 它不是一组 `after_agent` 中间件。

改造前问题：

- 默认反思中间件可额外调用模型。
- 每次任务自动把目标和最终回答写成长期记忆。
- 最终回答通过扫描最后一条 AI 消息取得。
- 用量从事件重新求和。
- 模型、MCP、子进程和临时密钥缺少统一清理所有者。

目标调整：

- 完成、等待、失败、取消、失去租约和关停都进入同一清理路径。
- 最终回答引用 P8/P9 指定的助手消息版本。
- 用量取自模型调用账本，工具结果取自工具回执。
- 关闭所有属于本次执行的外部资源后才能交给 P12。
- 统计从权威记录计算，不在结束时额外调用模型。
- 用户明确要求记住时通过工具写入；自动记忆提议放在任务完成之后。

完成标准：所有结束方式关闭相同资源集合；无法确认子进程静止时不释放作业给新代次；默认结束路径没有隐藏模型调用。

当前进度：**P11 已完成。** `build_agent()` 不再装配反思和自动记忆回写中间件，结束路径没有隐藏模型调用。完成、等待、普通失败、用户取消、守护进程关停和租约丢失都先退出同一个执行级 `AsyncExitStack`，再从持久助手草稿、模型调用账本、工具回执、产物引用和验证记录生成结构化执行摘要。最终回答不再扫描图消息，用量也不从 SSE 反推。

清理失败或租约过期时，Runtime 会把 Run 标为 `cleanup_required`，持久化 `run.cleanup_required`，并封存原作业的所有者和租约代次；该作业不能被新执行者领取，也不能作为普通重试、修复或分支的父任务。只有原执行者确认资源已经静止，才能把隔离任务结算为失败并释放作业。即使原执行者先于回收器发现租约过期，它也会先原子补写隔离和结果不明账本，再提交清理证明，不会丢失确认。守护进程关停会立即取消活动执行并等待相同清理路径；超时会拒绝继续关闭存储。`memory.write` 改为显式工具：能力只从主任务的真实用户输入生成；消息中的所有非空行都必须是直接、肯定的记忆指令，代码块、引用、否定和混合说明不会授权写入。子 Agent 永远不获得写权限，记忆读写与清空均按身份和工作区隔离。

优先级：**A/C**。

## P12：结果原子提交和客户端投影

**必要性：保留。** 这是运行时唯一的任务结果提交点。

当前问题：

- 已完成第一批基础改造：等待或终态状态与对应事件现在由同一事务提交。
- 已完成第一批基础改造：相同结果可安全重放，冲突终态会被拒绝；提交失败会整体回滚。
- 已完成第二批顺序收口：结果事务会同时结束旧执行作业，提交后才通知客户端；客户端立即审批或回答时可以可靠创建下一条恢复作业，不再受旧协程的内存清理窗口影响。
- 已完成第一批基础改造：事件序号写入已串行化，并增加 `(run_id, seq)` 唯一约束；旧重复序号会在启动时修复。
- 已完成租约边界收口：普通执行只能用属于当前任务、尚未过期且未隔离的租约提交结果；无租约提交默认拒绝，启动恢复流程必须显式进入孤儿恢复模式，并在事务内证明没有待执行或执行中的作业。
- 已完成运行时对话所有权迁移：客户端只消费事件并按快照重建投影，不再拥有终态和聊天正文。

目标调整：

- 一个受租约代次保护的事务提交完成、等待、重试、失败或取消。
- 同一事务更新任务、作业、执行尝试、助手消息和引用。
- 同一事务写任务事件和全局资源变化记录，提交后才通知客户端。
- 相同结果重复提交为无操作；冲突结果或旧代次直接拒绝。
- 客户端只维护可删除、可重建的投影缓存。
- 通知、索引、自动记忆和质量评估作为任务完成后的独立工作。

完成标准：

- 在事务每一步模拟崩溃都不会产生半终态。
- 客户端漏掉终态事件后重新读取快照仍能得到正确状态。
- 清空客户端数据后可从运行时完整重建对话。
- 失去租约的旧执行不能写任务、消息、事件或检查点。

优先级：**A**。

当前进度：**P12 已完成。** Runtime 已保存版本化对话和有序消息；接纳事务同时创建命令、消息、Run、作业和线程变化，结果事务同时更新 Run、助手消息、终态事件、线程版本与作业结算。普通执行提交前会在同一事务中复核租约所属任务、所有者、代次、期限和隔离状态；旧执行、串错任务的执行和过期执行都会整体回滚。无租约提交默认拒绝；只有启动恢复代码显式进入孤儿恢复模式，并在事务内确认不存在待执行或执行中的作业时，才能写入确定性结果。等待、取消、普通失败、完成、清理隔离和隔离解除都走同一投影边界。检查点分叉会创建新的产品对话，但复用原图线程的指定检查点，不会改写旧分支。

客户端把 SSE 当作即时增量和唤醒信号。即使漏掉终态事件，流结束后和后台游标轮询都会重新读取权威线程快照。线程列表使用不变的创建时间游标分页；变化读取会在有界范围内连续翻页到当时的尾部，再按每个线程的最后变化处理，避免更新与稍后的删除跨页时卡住；积压超过一万条就放弃增量并执行完整快照同步，不无限堆积内存。长对话按消息位置分页，并在后续页校验同一线程版本，读取期间有写入就整体重试。过程事件有明确上限并标记截断，正文消息仍可完整重建。Electron 的 IndexedDB 只保留可删除缓存；重命名、置顶、工作区元数据、删除、编辑重跑和检查点分叉都先提交 Runtime。非本机身份提交的客户端线程编号和助手消息物理编号都会映射到身份作用域，远程用户之间不能探测或抢占同名资源。

后续工作不再属于 P12 正确性缺口：远程认证和跨设备连接属于 P1；搜索索引、质量评估或通知若加入，必须作为结果提交后的独立持久作业，不能回滚用户任务。

## 推荐迁移顺序

1. **状态基础**：P2、P3、P4、P5、P12。先建立权威数据、命令幂等、作业租约和原子结果提交。
2. **运行时独立**：P1、P6、P8。完成 Desktop 只走 Runtime、BYOK 和无云端回退。
3. **图执行边界**：P7、P10、P11。修正恢复、工具副作用和资源清理。
4. **执行质量**：P8、P9。统一上下文预算、验证和完成判断。
5. **删除旧路径**：删除 Web Cloud Agent、Go 模型/工具强依赖、客户端双历史和旧中间件。
6. **远程客户端准备**：在同一协议上增加远程认证和能力发现；不复制 Harness。

不要按 P1-P12 的编号顺序直接写代码。阶段编号描述一次任务的运行顺序，迁移顺序由依赖关系决定。
