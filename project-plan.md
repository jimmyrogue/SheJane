# 简单 Jiandan — 产品计划书

**品牌名：** 简单 / Jiandan
**域名：** jiandanly.com
**Slogan：** 把复杂的工作，简单做完

> "让每个小团队都有自己的 AI 员工。不需要懂技术，打开就能用。"

---

## 一、项目概述

### 1.1 我们要做什么

**简单（Jiandan）** 是一款面向中小企业和职业人群的 AI 生产力工具。不是另一个 ChatGPT 客户端，不是给极客用的多模型管理器，而是一个**围绕工作场景组织的 AI 工具箱**——帮助教师、销售、行政、小企业主等非技术用户，用 AI 提升效率，自动化日常琐碎工作。

名字本身就是产品承诺：**简单**。用户不需要理解 AI 是什么，不需要配置任何东西，打开就能用。

### 1.2 核心理念

- **简单好用** — 下载即用，零配置，不需要理解模型、API、token 等概念
- **场景驱动** — 用户看到的是"帮我写 / 帮我读 / 帮我算"，不是空白对话框
- **包月为主** — 每月给固定额度，成本可预期；额度用完后可额外充值
- **默认提供模型** — 开箱即用，用户不需要准备 API key，也不需要理解供应商差异
- **本地优先数据** — 长期聊天历史默认保存在客户端，云端只处理账号、计费、模型调用和临时文件

### 1.3 一句话定位

> 比豆包专业，比 ChatBox 简单。中小企业的第一个 AI 生产力工具。

## 二、市场分析

### 2.1 市场规模与趋势

- 2025 年美国中小企业 AI 投入比例已达 57%，较 2023 年增长 58%
- 82% 的小企业雇主已投资 AI 工具，平均每家使用 5 种 AI 工具
- 93% 计划持续投入，62% 计划增加 AI 支出
- 中小企业用 AI 的最大痛点不是"没有工具"，而是"工具太分散、太碎片"

### 2.2 竞品格局

|------|------|------|------|
| Cherry Studio | 极客多模型工作站 | 功能全面，开源，40k+ Star | 配置复杂，面向技术用户 |
| ChatBox | 跨平台 AI 客户端 | 全平台覆盖，月入 $90万 | 偏技术向，纯订阅制，无团队版 |

- 比 ChatGPT Team 便宜 5 倍

- 比 Cherry Studio 简单 10 倍
- 面向中国市场有本地化优势（人民币定价、微信支付宝、国产模型优先）

## 三、目标用户

### 用户 A：小企业主 / 创业者

- 5-30 人公司，什么都管
- 高频需求：写客户邮件/方案、整理合同文档、简单数据分析、翻译
- 痛点：每天花 2-3 小时在"不直接赚钱但必须做"的事情上
- 付费意愿：强，月花 ¥50-200 完全接受

### 用户 B：职业人（教师 / 销售 / 行政 / HR）

- 企业内部员工，大量重复性文字/数据处理
- 高频需求：教案生成、客户跟进话术、会议纪要、招聘 JD、报表总结
- 付费意愿：个人中等（¥20-50/月），公司采买不敏感

### 用户 C：企业采购者（小团队负责人 / IT 负责人）

- 需要给团队 5-20 人统一提供 AI 能力
- 需求：统一管理、可控成本、使用报表
- 付费意愿：按人头/按量付费都行，需要发票

## 四、产品设计

### 4.1 核心场景

用户打开 app 看到的是任务卡片，而非空白对话框：

| 场景 | 功能 | 用户感知 |
|------|------|----------|
| 📝 帮我写 | 邮件、方案、公告、通知、周报 | 选类型 → 填关键信息 → 得到成品 |
| 📄 帮我读 | 拖入 PDF/Word → AI 总结/提取/翻译 | 拖文件 → 问问题 → 得到答案 |
| 📊 帮我算 | 拖入 Excel/CSV → AI 分析/图表 | 拖表格 → 看结论 |
| 🔄 帮我翻译 | 输入文本或拖入文件，多语言互译 | 粘贴 → 选语言 → 得到翻译 |
| 💬 自由对话 | 传统聊天模式，什么都能问 | 传统 AI 对话体验 |

### 4.2 模型策略：不让用户选模型，让用户选"效果"

- **快速模式（默认）** → 后端路由到 DeepSeek / Qwen 等高性价比模型，消耗少
- **深度模式** → 后端路由到 Claude Sonnet / GPT-4o 等强模型，消耗多但质量更好
- 用户感知是"普通回答"和"精细回答"的区别，类似出租车和专车
- 进阶用户可在设置中打开"手动选择模型"

### 4.3 团队版功能（后期利润核心）

团队版不进入早期实现。Phase 1-3 先验证个人用户的高频工作流、付费意愿和留存，再把稳定的个人能力包装成团队管理版本。

- **管理后台** — 查看团队总用量、每人用量、费用
- **统一充值** — 公司充一笔钱，分配给员工
- **共享知识库** — 上传公司资料，团队成员对话时自动引用
- **角色预设** — 管理员为不同岗位配置不同 AI 助手

### 4.4 未来方向（第二阶段）

- 定时任务 — 每天自动总结邮件/群消息
- 模板工作流 — "收到客户询价 → 自动生成报价单草稿"
- 外部集成 — 连接飞书/钉钉/企业微信

### 4.5 数据策略：Hybrid Local-first

早期对标 Chatbox 的混合模式：用户付费使用平台统一模型额度，但长期数据不默认进入云端。

- **客户端主存储**：聊天历史、对话列表、重命名、归档、搜索、导出默认在本地完成
- **云端临时处理**：模型请求会经过 Jiandan 后端，用于模型路由、额度扣减、供应商密钥保护和限流
- **后端最小留存**：后端默认只保存账号、订阅、钱包账本、调用 metadata、token / credits 用量、错误码和支付记录
- **文件按需上传**：文件只在需要模型处理时临时上传，默认设置过期时间；个人云文件库和云知识库后置
- **云同步 opt-in**：跨设备同步、云端历史、云知识库必须由用户主动开启，不能进入 Phase 1 MVP

## 五、商业模式

### 5.1 定价策略

| 版本 | 价格 | 包含内容 |
|------|------|----------|
| 免费试用 | ¥0 | 新用户赠送一次性体验额度，用完后升级或充值 |
| 个人专业版 | ¥39/月 | 每月固定对话额度，支持文档/数据分析等高级场景 |
| 团队版（后期） | ¥29/人/月（5人起） | 团队月额度池 + 成员上限 + 管理后台 + 发票 |
| 额外额度包 | ¥9.9 / ¥29 / ¥99 | 月额度用完后的补充额度，不自动续费 |

当前阶段不支持 BYOK（自带 API key）。这类能力如果未来需要，放到开放平台阶段重新评估。

### 5.2 月额度 + 额外额度机制

- 不暴露 token 概念，包装成"本月额度"和"额外额度"
- 每个订阅周期自动发放月额度；未用完的月额度默认不结转
- 额外额度包用于月额度耗尽后的继续使用，可设置较长有效期
- 请求前先预留额度，响应结束后按实际消耗结算，避免并发超用
- 展示"本月约可对话 N 次 / 额外额度约可用 N 次"而非"剩余 XX tokens"
- 新用户送免费额度，先体验到价值

### 5.3 收款链路

- **法律主体：** 美国公司（LLC / C-Corp）
- **支付平台：** Stripe（已有美国主体账户）
- **订阅支付：** 信用卡 + Apple Pay，作为包月主路径
- **额外额度包：** 信用卡 + Apple Pay + 支付宝 + 微信支付，作为一次性充值
- 支付宝 / 微信支付按一次性支付处理，不作为自动续费订阅的核心依赖
- 以人民币展示价格，Stripe 处理收款与结算；后台保存原始币种、汇率和手续费，方便对账

```
└───────────────────────┬─────────────────────────┘
                        │ OpenAI-compatible API

┌─────────────────────────────────────────────────┐
│        Go API Gateway (Auth / Billing / LLM)      │
│                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │Auth 中间件│ │模型路由器 │ │ SSE 流式代理     │  │
│  │JWT 鉴权   │ │快速/深度  │ │ 转发 LLM 响应    │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │额度账本  │ │限流器    │ │ Prompt 模板      │  │
│  │预留/结算 │ │个人优先  │ │ 场景注入         │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
│  ┌──────────────────────────────────────────────┐ │
│  │ 临时文件处理 / 重试 / 降级 / 超时 / 熔断         │ │
│  └──────────────────────────────────────────────┘ │
└───────────────────────┬─────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│           LLM Providers (对用户不可见)              │
│  DeepSeek │ Claude │ GPT │ Qwen │ Gemini │ ...  │
└─────────────────────────────────────────────────┘
```

### 6.2 技术栈

| 层 | 技术选型 | 说明 |
|----|----------|------|
| 桌面端 | Electron + React + TypeScript + Vite | 生态成熟，快速出 MVP，与 Web 端共享代码 |
| Web 端 | React（共享组件） | 桌面端和 Web 端共享核心 UI 组件 |
| 移动端 | PWA / Capacitor（第二阶段） | 先验证需求再决定是否做原生 |
| 后端 | Go (net/http + chi) | 单体二进制，高并发 SSE 流处理 |
| 数据库 | PostgreSQL 16 (自建) | 用户、计费、用量日志、分享快照、Prompt 模板；不默认存聊天正文 |
| 缓存 | Redis 7 | 限流、短期流式缓存、验证码；账务以 PostgreSQL 为准 |
| 认证 | Go 自建 (JWT + bcrypt + OAuth) | 灵活，方便后续接微信登录 |
| 文件存储 | AWS S3 香港 Region | 临时处理文件、分享附件、后期 opt-in 云文件库 |
| 支付 | Stripe (Card / Apple Pay 订阅 + Alipay / WeChat Pay 充值) | 已有美国主体账户 |

前端客户端的详细模块、数据流、Web/Electron 差异和 Agent 演进路线见 [`frontend-spec.md`](frontend-spec.md)。
后端的最小留存、调用记录、临时文件和分享快照边界见 [`backend-spec.md`](backend-spec.md)。

### 6.3 后端核心模块

**LLM 代理层：**
- 对外暴露 OpenAI 兼容的 `/v1/chat/completions` 接口
- SSE 流式代理，Go goroutine 逐请求转发
- 模型路由器：快速模式 → DeepSeek/Qwen，深度模式 → Claude/GPT
- 重试 + 降级：供应商限流时自动切换备选模型
- 默认不保存完整 prompt / response / messages，仅保存 request_id、模型、用量、状态和错误 metadata

**计费层：**
- 包月额度为主，额外充值额度为补充
- 请求前在 PostgreSQL 中创建额度预留，避免并发超用
- 响应结束后按实际 token / 工具成本结算，多退少补，写入账本
- Redis 只做缓存和限流，不作为余额真实来源

**团队管理层（后期）：**
- Organization → Teams → Users 数据模型
- 管理员：邀请成员、设置用量上限、查看报表、充值续费

**本地优先数据层：**
- Web 使用 IndexedDB，Electron 使用本地 SQLite / 文件数据库保存长期聊天历史
- 后端不提供 Phase 1/2 的完整云端 conversation CRUD；分享时才上传用户确认的快照
- 云同步、云端历史和云知识库是后期 opt-in 能力

**Prompt 模板层：**
- 每个场景（帮我写/读/算）对应预设 system prompt
- 后期团队管理员可自定义团队专属模板
- 模板存储在数据库，支持版本迭代

### 6.4 计费与合规边界

- 当前阶段不提供 BYOK，避免引入供应商密钥加密、免费资源滥用、客服排障等复杂度
- 文件上传、文档解析、Office 生成、图片生成、RAG 都会消耗平台资源，必须走统一额度账本
- 长期聊天历史默认留在客户端；只有用户主动分享、反馈问题或开启未来云同步时，才上传选定内容
- 面向中国用户时，香港部署不能等同于规避所有数据合规义务；隐私政策、跨境处理告知、删除/导出能力需要在 MVP 前补齐
- 所有支付回调、额度发放、额度扣减都必须有幂等键和审计日志

### 7.4 为什么选择香港

- 大陆用户访问延迟通常显著低于美国机房，适合早期验证体验
- 美国主体 + 香港 / 新加坡部署能降低早期运维复杂度，避免过早投入大陆备案和多区域架构
- 这只是部署和延迟折中，不等于规避隐私、跨境处理、供应商转发和删除/导出义务
- 早期采用 Local-first，减少云端长期存储的聊天正文和文件正文，降低合规与存储压力
- 如果后续进入团队版 / 企业版 / 大陆客户采购，再评估中国大陆独立部署或客户私有化部署

### 7.5 为什么不用 Supabase

- Supabase Cloud 在中国延迟高、连接不稳定
- Self-host Supabase 运维负担重（Kong + GoTrue + Realtime + Storage 等组件）
- 我们真正需要的只是 PostgreSQL + 自建 Auth
- Go 自建认证（JWT + bcrypt + OAuth）更灵活，方便后续接微信登录
- 一个 Go binary + 一个 PostgreSQL 实例 = 全部后端，运维极简

## 八、开发计划

### Phase 1：可收费聊天 MVP（3-4 周）

**目标：** 先做一个能注册、能聊天、能订阅、能稳定扣费的最小闭环。

- [ ] Go 后端：OpenAI 兼容 API + SSE 流式代理 + 模型路由（DeepSeek + Claude）
- [ ] Go 后端：JWT 认证 + 用户注册登录
- [ ] Go 后端：订阅计划 + 月额度发放 + 额度预留 / 结算账本
- [ ] Stripe 支付集成：信用卡 / Apple Pay 包月订阅 + Webhook 幂等处理
- [ ] Electron 桌面端：基础对话界面 + 快速/深度模式切换 + 本地聊天历史
- [ ] Web 端：共享组件，IndexedDB 本地聊天历史，直连后端模型网关
- [ ] 本地数据能力：对话导出 / 导入、删除、基础搜索；不做云端自动同步
- [ ] 前端规格落地：按 [`frontend-spec.md`](frontend-spec.md) 实现 Phase 1 客户端边界
- [ ] AWS 香港部署：Docker Compose 一键启动

### Phase 2：场景化工作台（3-5 周）

- [ ] 场景卡片首页（帮我写 / 帮我读 / 帮我翻译）
- [ ] 文档上传 + AI 分析（PDF / Word / Excel）
- [ ] Prompt 模板系统
- [ ] 本地对话搜索 / 导出增强；分享由用户主动上传快照，支持撤销和过期
- [ ] 额外额度包：信用卡 + Apple Pay + 支付宝 + 微信支付一次性充值

### Phase 3：个人高级能力（4-6 周）

- [ ] 个人本地文件库：本地文件引用、上传历史、文件状态、再次引用
- [ ] 个人本地知识库（轻量 RAG）：本地索引优先，云端知识库仅作为后期 opt-in
- [ ] 个人 Prompt 收藏和常用场景快捷入口
- [ ] 用量历史：按日 / 按场景查看个人消耗
- [ ] 个人设置：通知偏好、数据导入 / 导出、账号安全、云同步开关预留但默认关闭

### Phase 4：个人 Agent 与生成任务（6-8 周）

- [ ] Tool Calling / Agentic Loop，但所有工具通过异步 job 队列执行
- [ ] API + 文件工具 Agent：围绕个人文件、对话和生成任务执行
- [ ] Office 文件生成（Excel / Word / PPT）+ 文件大小 / 轮数 / 耗时限制
- [ ] 图片生成三档模型 + 异步任务状态

## 九、竞争壁垒

1. **品味和体验打磨** — 普通用户对"好不好用"的敏感度远高于对"功能多不多"的敏感度
2. **内容运营** — 作为内容创作者，通过实操视频和场景展示持续获客，这是纯技术产品做不到的
3. **中国市场本地化** — 人民币定价、微信支付宝、国产模型优先、符合国内用户习惯
4. **团队版粘性** — 企业采买后，切换成本高，续费率高
5. **成本控制** — 规模上来后与供应商谈更优惠的价格，接入高性价比国产模型拉低成本
